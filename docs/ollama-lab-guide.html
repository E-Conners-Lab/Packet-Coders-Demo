01

The shape of the thing

One idea unlocks all of this: the model never speaks MCP. A host runs the tool-calling loop and connects typed tools to whatever model you point it at. Swap the model freely.

host

Open WebUI

Your chat UI. Runs the tool-calling loop. Talks to tools via mcpo, and to a model for completions.

  • mcpo → packet-coders-mcp
  • chat completions → backend ↘
inference

Local model backend

Serves chat completions. Pick either — the host doesn't care which.

  • Ollama @ :11434 — easy, great default
  • vLLM @ :8000/v1 — OpenAI-compatible, GPU
tools

packet-coders-mcp  ·  six typed Python functions

MCP tools are just functions with good docstrings. The networking stays in a normal service layer; FastMCP is deliberately thin.

  • list_lab_devices
  • send_command (read-only)
  • run_health_check
  • get_ospf_neighbors
  • get_bgp_summary
  • configure_device (gated)

03

Build it — follow along

Steps 1–3 need nothing but a laptop (no lab, no GPU, no Tailscale). Steps 4–6 add the chat UI. Hit the copy button on any block.

1
Pull a small tool-calling model ~2 GB · needs ~8 GB RAM · runs entirely on CPU
$ ollama pull llama3.2
2
Install the project FastMCP server + the smoke-test harness, into a local venv
$ uv venv # create the environment
$ uv pip install -e .
3
Smoke-test the whole loop A local model calls a real tool against the built-in mock lab. This is the money shot.
LLM_MODEL=llama3.2 uv run python scripts/local_llm_smoke.py "List the lab devices."
✓ expect The model calls list_lab_devices and answers r1, r2, spine1 — all local, no cloud, no network gear. Read-only tools only (no configure_device), so it can never write.
4
Expose the tools through mcpo Wraps the stdio MCP server as an OpenAPI tool server Open WebUI can call
PACKET_CODERS_INVENTORY=<REPO>/configs/inventory.mock.yaml \
  uvx mcpo --port 8000 -- uv run --project <REPO> packet-coders-mcp
→ then In Open WebUI: Settings → Tools → add tool server http://localhost:8000. The six lab tools now show up to the model.
5
Connect the Ollama backend Point Open WebUI at Ollama — local, or over Tailscale to a beefier box
# Open WebUI → Settings → Connections → Ollama API
# local:     http://localhost:11434
# tailnet:   http://<your-tailnet-host>:11434
★ tip Use a tool-calling-capable model — Qwen3 is a strong pick; it selects tools reliably and emits clean JSON. Keep smaller/Gemma models on read-only tools.
6
Ask it something real Now chat. The model reaches for the tools on its own.
› Are SW1's OSPF neighbors all FULL?
  ↳ calls get_ospf_neighbors(device="SW1")
  ↳ "Yes — SW1 holds FULL with SW2, SW3, SW4 (3/3)."

Scale up — one variable at a time

The 5-minute loop is the whole thing. Everything beyond it is changing exactly one knob. Never more than one — that's how you keep failures debuggable.

Better tool-calling
ollama pull qwen3:8b LLM_MODEL=qwen3:8b
A different / remote model
LLM_BASE_URL=http://<host>:11434/v1 # Ollama LLM_BASE_URL=http://<host>:8000/v1 # vLLM
A real lab instead of mock
PACKET_CODERS_INVENTORY=./inventory.local.yaml
Add an external connector
uvx mcpo --config ~/.config/...mcpo.config.json # adds /filesystem alongside /packet-coders
05

The traps that cost me an evening

Every one of these looks like "the model is broken." None of them are. Show these on screen — they're the difference between a demo that works and an hour of head-scratching.

docker urls Tool server ≠ model connection (Open WebUI in Docker)

Open WebUI calls tool servers from your browser, but reaches the model backend from inside the container. Same machine, two different URLs — host.docker.internal for one, localhost for the other.

✓ Ollama — backend

Settings → Connections · http://host.docker.internal:11434

✓ mcpo tools — browser

Settings → Integrations · http://localhost:8000

Native / pip Open WebUI (not Docker)? Both are just localhost.

function calling Tools connected, but the model never calls them

An Ollama model in Open WebUI ignores connected tools unless function calling is set to Native. The tool server shows green; the model just never fires it.

Fix: model Advanced Params → Function Calling → Native (qwen3 supports it). Pair with the raised context window above.

connection type vLLM is not an Ollama connection

Add Ollama under Ollama API; add vLLM under OpenAI API. They are separate sections with separate master toggles. Put vLLM in the wrong one and Open WebUI polls /api/tags (which vLLM doesn't serve) — the model silently never appears.

✓ Ollama

section: Ollama API · URL http://<host>:11434 · no /v1

✓ vLLM

section: OpenAI API · URL http://<host>:8000/v1 · key = any non-empty string

If only Ollama models show: the global OpenAI API toggle is off — flip it on, then hard-refresh.

context window "The model ignored the tools"

Ollama defaults to a tiny context (~4K). The tool schemas plus verbose show output overflow it, so the model never sees the tools and just makes something up.

Fix: raise it — OLLAMA_CONTEXT_LENGTH=32768 ollama serve, or bake num_ctx into a Modelfile.

vLLM tool parser Tool calls come out as plain text

Tool calling is off by default in vLLM. Without the right parser the model emits tool calls as prose and nothing fires.

vllm serve <qwen3-model> --enable-auto-tool-choice --tool-call-parser hermes
Note: the parser is model-specific (Qwen-family commonly uses hermes) — verify against current vLLM docs for your exact model.

URL shape Trailing /models on the base URL

The OpenAI base URL ends in /v1 — never /v1/models. Open WebUI appends /models itself, so the extra one makes it query /v1/models/models → empty list.

06

Keep a human in the loop

This is a demo server, not a production change platform — but the guardrails are real, and they're the part worth narrating on camera.

🔒Reads are open, writes are gated. send_command blocks config & destructive commands.
🔒Writes are gated, not off. configure_device is available by default, but every change needs the one-time console code; PACKET_CODERS_ALLOW_WRITES=false hides it entirely for read-only.
🔒Hard-blocked lines. reload, erase, delete, write erase never run.
🔒The model can't self-approve. A real change needs a one-time code printed only to the server console — a human reads it and supplies it, even in an auto-executing UI.
🔒Backends stay on the tailnet. An open Ollama/vLLM port is an unauthenticated model and tool surface.
🔒Secrets stay local. Real IPs & creds live only in git-ignored inventory.local.yaml.
▣

Suggested webinar flow

  1. Start in mock mode — list devices, no lab required.
  2. Show server.py: how @mcp.tool turns plain Python into MCP tools.
  3. Run get_ospf_neighbors & get_bgp_summary against the mock lab.
  4. Switch the inventory to the EVE-NG four-switch mesh — same tools, real adjacencies.
  5. Demo configure_device: the model proposes a change, you read the one-time code off the server console to approve it — add & remove a demo loopback, watch it ride eBGP.
  6. Repeat the whole thing driven by a local Qwen3 model — same tools, zero cloud API.
