The hook

You've already asked ChatGPT
about a down BGP session.

It gave a great answer. But notice: you were the hands — you copied output in, read the fix, typed it back at the CLI.

The model is smart, but it's behind glass. It can't see your network, and it can't touch it. MCP takes the glass away.

The ceiling of plain chat

Three things chat can't do

  • 01No live data.It knows networking in general — not your SW3's neighbor table at 9am today.
  • 02No actions.It can write a perfect router bgp block — it can't apply it. No hands on the keyboard.
  • 03You are the integration.Copy from switch → paste to chat → read → paste back. Every manual loop is where mistakes live.

"I'll just write a script." Sure — now you maintain regex parsers. MCP is the middle path: flexible like chat, with safe, reusable actions.

What MCP actually is

USB-C for AI tools.

Before USB-C, every device had its own connector. Before MCP, every AI app wired up tools its own way.

MCP is the standard plug. Build a tool once — any MCP-aware app can use it. You're not building a brittle one-off; you're publishing a capability.

The one idea that unlocks all of it

Three roles — and the model never speaks MCP.

tools

MCP Server

Typed Python functions that know how to SSH to your switches and run commands.

app

Host

Claude Desktop / Code / Open WebUI. Runs the tool-calling loop and does the plumbing.

brain

Model

Claude or local Qwen. Just sees "here are tools" and picks one. Never touches the protocol.

Because they're separate, you can swap any piece. Same tools work with a cloud model or a fully local one. Add a tool → every host gets it for free.

A tool is just a function

This is the entire definition of a tool.

@mcp.tool async def get_bgp_summary(device_name: str) -> dict: """Read BGP summary from a lab device using the right command for its platform.""" return lab.get_bgp_summary(device_name)

The docstring is what the model reads to decide when to call it. Writing tools is mostly writing good documentation. The networking lives in a normal service layer behind it.

Why this fits networking like a glove

Our day is already tool-shaped.

Go look at a thing → interpret it → maybe change one thing → verify. That's exactly what a tool-calling model is good at.

  • › "Are all my OSPF adjacencies up?"
  • › "Which BGP sessions are flapping?"
  • › "Did that change propagate to the other switches?"

Ask once. It fans out across every device, reads the right command per platform — EOS vs IOS vs Junos — and gives you the synthesized answer.

The line that makes it usable on real gear

Reads open. Writes gated.

  • ✓The model can look at everything, all day.
  • 🔒To change anything, a human approves — and the model can't approve its own change.
  • 🔒The confirm code prints to the server console only — never back to the model.
The part everyone asks about

Standard MCP  vs.  MCPo

Not two sets of tools. Two doorways to the same ones.

Doorway 1 — the native protocol

Standard MCP

  • ›The native protocol. Claude Desktop & Claude Code speak it directly.
  • ›stdio — host launches the tool server as a local subprocess, talks over stdin/stdout.
  • ›or HTTP for remote servers. Either way it's session-based — a live conversation.

If your client speaks MCP, this is all you need. Cleanest path, fewest moving parts.

Doorway 2 — the bridge

MCPo = MCP → OpenAPI

Lots of apps — Open WebUI especially — don't speak MCP. They speak OpenAPI (REST/Swagger), which nearly everything understands.

  • +Any OpenAPI app can use your tools — not just MCP clients.You rewrite nothing — same Python functions; mcpo just translates.
  • +Free interactive docs — a /docs page, "try it" in a browser, or curl it.
  • +Standard HTTP — API-key auth, logging, reverse proxy.
  • +Many servers, one port — lab tools at /lab, filesystem at /filesystem.
Which doorway, when

Same tools. Pick the door.

The demo.

Everything from here, I type in plain English. The model picks the tools.

switch to Claude Desktop / Code now
DEMO 1 · list_lab_devices

"What's even in this lab?"

you › What devices are in my lab?
SW1 arista_eos ssh 10.255.0.1 (65001) SW2 arista_eos ssh 10.255.0.2 (65002) SW3 arista_eos ssh 10.255.0.3 (65003) SW4 arista_eos ssh 10.255.0.4 (65004)

I never named a command or an IP — it picked the tool on its own.

DEMO 2 · run_health_check

"Is the control plane healthy?"

you › Run a health check across the whole lab — is OSPF and BGP healthy everywhere?
✓ SW1 healthy OSPF 3 · BGP 3 ✓ SW2 healthy OSPF 3 · BGP 3 ✓ SW3 healthy OSPF 3 · BGP 3 ✓ SW4 healthy OSPF 3 · BGP 3

One sentence → all four switches, concurrently, right commands per platform, summarized. That's a round of SSH sessions, gone.

DEMO 3 · get_ospf_neighbors + get_bgp_summary

"Prove SW1's adjacencies."

you › Show me SW1's OSPF neighbors and BGP summary. Are they all up?
OSPF 10.255.0.2 FULL 10.12.12.2 Et1 10.255.0.3 FULL 10.13.13.3 Et2 10.255.0.4 FULL 10.14.14.4 Et3 BGP 65002 10.12.12.2 Estab 3/3 65003 10.13.13.3 Estab 3/3 65004 10.14.14.4 Estab 3/3

Two tools, one question — read back in English, not columns.

DEMO 4 · send_command (read-only)

"A targeted question."

you › On SW3, what's the route to the other switches' loopbacks?
B 10.255.0.1/32 via 10.13.13.1 (eBGP) B 10.255.0.2/32 via 10.23.23.2 (eBGP) B 10.255.0.4/32 via 10.34.34.4 (eBGP)

The catch-all read tool blocks config & destructive commands at the door. It can look; it can't break.

DEMO 5 · configure_device · human-gated ★

A real change — with a human gate.

you › Add loopback 99 (10.99.99.99/32) on SW1, advertise it into OSPF, and confirm the others learn it.
  • 1Model proposes the config — and stops. A one-time code prints to my server console only.
  • 2I read the code off the console and hand it back. The model never sees it — it can't self-approve.
  • 3Applied. Now verify it propagated.
DEMO 5 · verify & clean up

Watch it propagate.

you › Does SW2 have a route to 10.99.99.99 now?
O 10.99.99.99/32 via 10.12.12.1 (OSPF, from SW1) ✓ learned
you › Now remove loopback 99 from SW1. → same gate, confirm, gone

Leave the lab as I found it. Optional flex: this whole demo also runs on a local Qwen via Open WebUI + mcpo — zero cloud.

Where to start

Don't boil the ocean.
Start read-only, on a lab.

  • →Chat is smart but behind glass. MCP gives the model tools.
  • →Standard plug: build once, use everywhere — Desktop, Code, or local via mcpo.
  • →A human approves every change. That's what makes it real.

It's open source — clone it, point it at your own lab, and you'll have a local model reading your network in ~5 minutes.
